⚡ Signals como Reactive Consumers

El mismo componente hijo con OnPush — tres formas de actualizar el padre y cómo cada una afecta (o no) al hijo.

A Mutación directa del objeto
❌ hijo stale
El padre muta user.score++ directamente. La referencia del objeto es la misma → Angular OnPush no detecta ningún cambio → el hijo no se re-renderiza.
Padre sabe: userA.score = {{ userA.score }} userA.level = {{ userA.level }}
B Spread operator — nueva referencia
✅ hijo actualiza
El padre crea userB = {{ "{ ...userB, score: n+1 }" }}. Nueva referencia → Angular detecta el cambio de @Input → el hijo OnPush sí re-renderiza.
Padre sabe: userB.score = {{ userB.score }} userB.level = {{ userB.level }}
C input() signal — API moderna
⚡ reactivo
input() declara el @Input como un Signal. El hijo puede derivar computed() y effect() directamente del input — sin ngOnChanges ni SimpleChanges. El template es un reactive consumer nativo.
Padre pasa: userSignal() = score {{ userSignal().score }}, level {{ userSignal().level }}
